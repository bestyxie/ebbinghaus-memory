# ExecPlan 规范

ExecPlan 是本项目的执行计划文档。一个无状态的智能体——或一个人类新手——仅凭一份 ExecPlan 就能产出可工作的、可验证的结果。

计划（Plan）和任务（Task）是两个层级。本文档管"如何渐进式地推进计划"。拿到一份计划后如何拆分和执行任务，见 `docs/exec-plans/TASKS.md`。

## 核心方法论：渐进式迭代

**给定一个宏观目标，如何一步步把它变成现实？**

不试图一次规划到底。每次只定"下一个可交付的增量"，做完后站回全局视角再定下一个。

```
回顾历史 → 审视现状 → 对照全局目标 → 推导出下一个最小可交付增量 → 执行 → 验证 → 循环
```

### 操作步骤

1. **回顾**：通读 `completed/` 下所有已完成计划的 `plan-summary.md`（按序号从小到大），理解系统真实现状，识别之前的决策是否仍合理。
2. **定位**：明确全局目标（终点在哪）和当前现状（已走到哪），从差距中找到当前最值得做、最可行的一个切片。
3. **写计划**：只为这个切片创建一个计划目录。范围小到一个工作会话内能完成并验证。
4. **执行、验证、归档**：按 `TASKS.md` 的方法拆分任务并执行。完成后写 `plan-summary.md`，将目录移入 `completed/`。
5. **重新站到全局视角**：再次回顾所有已完成计划的 summary，决定下一份计划。如果发现之前方向有偏差，新计划必须明确指出偏差并修正。

**禁止在一份计划中规划从零到完成的完整路径。**

**发现偏差必须修正，不可将就。** 回顾时如果发现之前已完成的计划存在方向错误、决策失误或产出缺陷，不要在新计划中绕过或默默修补——必须创建一份专门的修正计划，在 Purpose 中明确指出哪份计划的什么决策出了问题、为什么错、如何修正。修正计划与普通计划遵循同样的规范，但其存在的首要目的是纠偏，不是推进新功能。

## 目录结构

每份计划是一个目录，不是单文件。

```
active/（或 completed/）
└── 001-bridge-cli-scaffold/
    ├── plan.md              # 原始计划（完整的 ExecPlan 文档）
    ├── plan-summary.md      # 计划完成后的精炼总结（仅 completed 有）
    ├── tasks.md             # 任务调度台（状态总览 + 已完成摘要 + 下一个待执行 task id）
    └── tasks/               # 每个 task 的详情文件
        ├── 001-contract-layer.md
        ├── 002-client-layer.md
        └── ...
```

**命名格式**：`<三位序号>-<短横线描述>/`，如 `001-bridge-cli-scaffold/`。

序号在 `active/` 和 `completed/` 之间全局递增。新建时取两个目录最大序号 +1。

**生命周期**：新建 → `active/<序号-描述>/` → 完成后写 `plan-summary.md` → 整个目录移入 `completed/`。

## 各文件职责

### plan.md

完整的 ExecPlan 文档。格式和活文档章节见下方「plan.md 编写指南」和「骨架」。是计划的权威来源，包含所有上下文、设计决策、验收标准。

### plan-summary.md

**仅在计划完成、移入 completed 时编写。** active 阶段不存在此文件。

目的：让后续计划回顾时只读这个文件就够了，不需要翻阅完整的 plan.md。

必须包含：

- 一句话概括本计划做了什么
- 关键产出（新增了哪些包/模块/文件，改了什么）
- 关键决策及其原因（从 plan.md 的 Decision Log 中提炼）
- 遗留问题或后续建议
- 最终验证状态

篇幅控制在 50 行以内。

### tasks.md

任务调度台。详见 `TASKS.md` 规范。

### tasks/ 目录

每个 task 的详情文件。详见 `TASKS.md` 规范。

## plan.md 编写指南

### Purpose 先行

先写 `Purpose / Big Picture`：全局目标 → 当前现状 → 本计划范围。让读者三句话内知道"为什么做"和"做完后用户能看到什么"。

### 可观察结果为锚点

验收条件是人类可验证的行为（"运行 X 命令，看到 Y 输出"），不是内部属性（"添加了一个 struct"）。

### 显式指定仓库上下文

用仓库根相对路径命名文件，精确命名函数和模块。运行命令时写明工作目录。

### 验证不可选

每份计划必须包含：要运行的测试命令、预期输出、端到端验证场景。

### 幂等和安全

步骤可多次运行无副作用。如有风险操作，说明回滚方案。

### 捕获证据

关键的终端输出、diff、日志作为缩进示例包含在计划中。

## 不可妥协的要求

- **自包含**：当前形态包含新手成功所需的全部知识和指令。不说"如前所述"或"根据架构文档"。需要的知识就地嵌入。
- **活文档**：随进展、发现和决策不断修订。每次修订仍保持自包含。
- **小步增量**：范围是一个可独立交付、独立验证的增量。
- **可演示的结果**：产出可观察的工作行为，而非仅"满足定义"的代码。
- **平实语言**：每个专业术语要么立即定义，要么不用。
- **偏差即修正**：发现已完成计划有方向错误或决策失误时，必须创建专门的修正计划。禁止将就、绕过或默默修补。

## 格式

需要展示代码、命令或输出时，使用 4 空格缩进块（不嵌套三反引号围栏）。叙事性章节以散文为主。

## plan.md 活文档章节

以下章节不可省略，必须随工作进展持续更新：

- **Progress**：复选框列表，带时间戳，反映实际当前状态。
- **Surprises & Discoveries**：意外行为、bug、性能发现。附简洁证据。
- **Decision Log**：每个决策记录 Decision / Rationale / Date。中途改方向必须记录。
- **Outcomes & Retrospective**：完成时总结成果、差距、经验教训。对比原始目标。

## plan.md 骨架

    # <简短的面向行动的描述>

    本 ExecPlan 是活文档。Progress、Surprises & Discoveries、Decision Log、
    Outcomes & Retrospective 章节必须随工作进展保持更新。

    本文档遵循 docs/exec-plans/PLANS.md 规范。
    任务拆分遵循 docs/exec-plans/TASKS.md 规范。

    ## Purpose / Big Picture

    全局目标：……
    当前现状：……（总结相关已完成计划的 plan-summary.md 关键内容）
    本计划范围：从差距中推导出的下一个最小可交付增量。

    如果之前的计划存在方向偏差，在此明确指出并说明修正策略。

    ## Progress

    - [x] (2025-10-01 13:00Z) 已完成步骤示例。
    - [ ] 未完成步骤示例。

    ## Surprises & Discoveries

    - Observation: …
      Evidence: …

    ## Decision Log

    - Decision: …
      Rationale: …
      Date/Author: …

    ## Outcomes & Retrospective

    成果、差距、经验教训。对比原始目标。

    ## Context and Orientation

    当前状态描述（假设读者一无所知）。关键文件完整路径。术语定义。
    总结相关已完成计划（目录名 + plan-summary 关键产出）。
    如发现之前决策需修正，在此说明。

    ## Plan of Work

    散文描述编辑和添加的顺序。命名文件、函数、模块。

    ## Validation and Acceptance

    验收行为：具体输入 → 具体输出。测试命令 + 预期结果。

    ## Idempotence and Recovery

    哪些步骤可安全重复。风险操作的回滚路径。

    ## Interfaces and Dependencies

    要使用的库/模块/服务及原因。
    里程碑结束时必须存在的类型和函数签名。
