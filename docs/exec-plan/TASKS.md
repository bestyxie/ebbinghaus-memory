# 任务拆分规范

本文档管"拿到一份 ExecPlan 后，如何把它拆成可执行的任务并推进"。计划层级的方法论见 `docs/exec-plans/PLANS.md`。

两个层级共享同一个底层循环：

```
回顾已完成 → 审视当前状态 → 对照目标 → 推导下一个最小可交付增量 → 执行 → 验证 → 循环
```

## 文件结构

一个计划目录下，任务相关的文件：

```
001-bridge-cli-scaffold/
├── tasks.md              # 任务调度台
└── tasks/                # 每个 task 的详情
    ├── 001-contract-layer.md
    ├── 002-client-layer.md
    └── ...
```

### tasks.md（调度台）

所有任务的中枢。包含三个部分，按以下顺序排列：

    # Tasks

    ## 已完成任务摘要

    对所有已完成 task 的产出做一段精炼总结（不是逐个罗列，
    而是站在当前视角概括"系统现在到了什么状态"）。
    每完成一个 task 后更新此摘要。

    ## 下一个待执行任务

    **当前**: T3 (001-contract-layer)
    或
    **当前**: 无（需要拆出新 task）

    ## 任务列表

    - [x] T1: contract 层协议定义 — tasks/001-contract-layer.md
    - [x] T2: client 层实现 — tasks/002-client-layer.md
    - [ ] T3: server 层实现 *(in-progress)* — tasks/003-server-layer.md
    - [ ] T4: web 接入 — tasks/004-web-integration.md

`[x]` = done，`[ ]` = pending / in-progress / dropped。
in-progress 用 `*(in-progress)*` 标注，dropped 用 `*(dropped)*` 标注。

**同一时刻最多一个 task 处于 `in-progress`。**

### tasks/\<序号>-\<描述>.md（任务详情）

每个 task 是一个独立文件，格式固定：

    # T<ID>: <一句话描述>

    **Status**: pending | in-progress | done | dropped
    **Started**: —
    **Completed**: —

    ## 目标

    做完后系统会多出什么可观察的变化。

    ## 涉及文件

    - `packages/bridge/src/contract/methods.ts` — 新增 ping method schema
    - `packages/bridge/src/contract/__tests__/methods.unit.test.ts` — 新增测试

    ## 验证方式

    做完后运行什么命令、期望看到什么输出。

    ## 执行记录

    （执行过程中的关键步骤、遇到的问题、做的决策。实时记录。）

    ## 产出摘要

    （完成后填写。实际做了什么、改了哪些文件、遇到什么意外。）

字段说明：

| 字段                    | 规则                                                 |
| ----------------------- | ---------------------------------------------------- |
| **ID**                  | T1, T2, T3...。与 tasks.md 任务列表中的 ID 对应      |
| **文件序号**            | 三位数字，如 `001-`。按创建顺序递增                  |
| **Status**              | 四种状态，同时在 task 文件和 tasks.md 列表中保持同步 |
| **Started / Completed** | 时间戳。pending 时为 `—`                             |
| **目标**                | 人类可验证的结果，不是实现描述                       |
| **涉及文件**            | 预期要创建或修改的文件，完整路径                     |
| **验证方式**            | 具体命令 + 预期输出                                  |
| **执行记录**            | 实时记录执行中的关键信息，保留上下文                 |
| **产出摘要**            | 任务完成后回填，供 tasks.md 的"已完成任务摘要"提炼用 |

## 核心原则

**任务也是渐进式的。** 拿到一份 ExecPlan 后，不要一次把所有任务都拆完。第一个 task 的执行结果会改变你对后续 task 的理解。先只拆出当前最清晰、最可行的下一个 task，做完后再决定再下一个。

**发现偏差必须用新 task 修正。** 执行中如果发现之前已完成的 task 存在产出缺陷、方向错误或遗漏，不要在当前 task 中顺手修补——创建一个专门的修正 task，在其「目标」中明确指出哪个 task 的什么产出有问题、为什么错、如何修正。修正 task 与普通 task 遵循同样的格式，但其首要目的是纠偏。同时在 plan.md 的 Decision Log 中记录偏差原因。

## 执行循环

每次准备开始工作时，执行以下步骤：

### 1. 打开 tasks.md，读取现状

看「下一个待执行任务」和「已完成任务摘要」。

### 2. 有 in-progress 的 task？

打开对应的 task 文件，继续执行。

### 3. 没有 in-progress？回顾后决定下一个

1. **读 tasks.md 的已完成任务摘要**——当前系统实际到了什么状态。
2. **读 plan.md 的 Surprises & Discoveries**——有没有影响后续拆分的新发现。
3. **对照 plan.md 的 Plan of Work**——原计划的下一步是否仍然合理。

然后做出判断：

- **已完成的 task 有偏差** → 创建专门的修正 task（不是在下一个 task 中顺手改），优先执行。
- **有合理的 pending task** → 标为 `in-progress`，开始。
- **pending task 不合理** → 在 plan.md Decision Log 记录原因，标为 `dropped`，拆新 task。
- **没有 pending task** → 根据 Plan of Work 和当前现状拆出新 task（创建文件 + 更新 tasks.md 列表），标为 `in-progress`。

更新 tasks.md 的「下一个待执行任务」。

### 4. 执行完毕

1. 运行验证命令，确认通过。
2. 在 task 文件中：status 改 `done`，填 Completed 时间戳和产出摘要。
3. 在 tasks.md 中：更新列表状态为 `done`，更新「已完成任务摘要」，清空「下一个待执行任务」（或直接指向下一个）。
4. 在 plan.md 中：勾选 Progress 对应条目；如有意外发现写入 Surprises & Discoveries。
5. **回到步骤 3**（回顾后决定下一个）。

### 5. 所有任务完成

1. 运行 plan.md 中 Validation and Acceptance 的端到端验证。
2. 填写 plan.md 的 Outcomes & Retrospective。
3. 编写 plan-summary.md。
4. 将整个计划目录从 `active/` 移入 `completed/`。

## 拆分方法

### 识别切入点

阅读 plan.md 的 `Purpose / Big Picture` 和 `Plan of Work`，问自己：

- 什么是当前能做的最小的、能独立验证的变更？
- 它不依赖尚未存在的代码吗？
- 做完后能用一条命令或一个操作证明它是对的吗？

这就是第一个 task。

### 纵向切片，而非横向分层

❌ 按技术层横切：先写所有类型定义 → 再写所有实现 → 再写所有测试。

✅ 按功能纵切：选一个最小的端到端路径，从类型定义到实现到测试一次做通。

每一刀切下去，切片内部是完整的纵向贯穿（类型 + 实现 + 测试），切片之间是渐进式推进。

## 粒度标准

| 维度           | 标准                                             |
| -------------- | ------------------------------------------------ |
| **可独立验证** | 做完后能运行测试或命令证明正确，不依赖后续 task  |
| **原子性**     | 要么完整完成，要么可以安全丢弃不影响系统         |
| **可观察结果** | 产出是人类可看到的变化，不是"代码已写但还不能跑" |
| **时间盒**     | 一个专注时段内能完成                             |

## Task 内的执行节奏

进入一个 task 后：

1. **写最小可编译的变更**（一个函数签名、一个类型定义）
2. **立即验证**（typecheck、lint、单测）
3. **通过后才前进**
4. 关键步骤记入 task 文件的「执行记录」

每一步都保持系统处于可工作状态。

## 常见反模式

| 反模式                  | 问题                                                       | 正确做法                                                  |
| ----------------------- | ---------------------------------------------------------- | --------------------------------------------------------- |
| 一次拆完所有 task       | 后续 task 基于猜测，执行时频繁失效                         | 只拆下一个，做完再拆                                      |
| 按层横切                | 每个 task 结束时没有可验证结果                             | 纵向切片                                                  |
| task 之间不回顾         | 沿着错误路径前进                                           | 每完成一个 task 都重新审视                                |
| task 内攒一堆改动再验证 | 出错时难以定位                                             | 最小变更 → 立即验证                                       |
| 拆得太细                | 切换成本高于执行成本                                       | 保持纵向切片完整性                                        |
| 不填产出摘要            | 下次回顾时无法基于事实判断                                 | done 时必须填写                                           |
| tasks.md 摘要不更新     | 回顾时看不到系统真实状态                                   | 每完成一个 task 立即更新                                  |
| 发现偏差顺手修          | 修复范围不清晰，产出摘要混乱，回顾时无法区分"推进"和"修正" | 创建专门的修正 task，明确指出纠正的是哪个 task 的什么问题 |
